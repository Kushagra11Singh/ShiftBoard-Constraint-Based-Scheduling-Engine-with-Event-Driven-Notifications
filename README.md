# ShiftBoard

ShiftBoard is a staff scheduling engine that takes the pain out of building conflict-free rotas. You give it your staff, their skills, and the shifts you need covered — it figures out who goes where. Assignments are generated by a backtracking constraint satisfaction solver that checks every hard rule before it commits to anything: skill requirements, weekly hour caps, and no double-booking across overlapping shifts on the same day.

When the scheduler makes assignments, it fires events onto an Apache Kafka topic. A separate notification service picks those up and dispatches the alerts. A background worker polls for unscheduled shifts on a configurable interval so you don't have to trigger things manually. The whole system is containerised, observable via Prometheus, and tested in GitHub Actions on every push.

## Scheduling Algorithm

The solver lives in `scheduling-service/app/services/scheduler.py` and works as a backtracking search with forward checking.

Each shift can require more than one staff member, so the algorithm treats every `(shift, required_slot)` pair as a separate variable. The domain of each variable is the set of active staff members who are eligible for that slot. Before trying an assignment it runs three constraint checks:

1. **Skill match** — the staff member must hold every skill the shift requires
2. **No double-booking** — the staff member cannot already be assigned to another shift that overlaps on the same day
3. **Weekly hours cap** — adding this shift must not push the staff member past their contracted weekly maximum

After each assignment the solver prunes the domains of all remaining unassigned slots. If any domain becomes empty, it backtracks immediately rather than continuing down a dead-end branch. In practice this keeps runtime well under 200 ms even for large inputs.

## Quick Start

You need Docker Desktop installed and running. Nothing else.

```bash
git clone https://github.com/your-username/shiftboard.git
cd shiftboard
cp .env.example .env
docker compose up --build
```

Give it about 30 seconds for Kafka to finish initialising. Once you see these lines you are good to go:

```
shiftboard-api           | INFO [app.main] Application startup complete.
shiftboard-notifications | INFO [app.consumer] Kafka consumer ready – polling for events
shiftboard-worker        | INFO [app.worker] Worker loop started – interval=60s
```

Check that everything came up healthy:

```bash
curl http://localhost:8000/health
curl http://localhost:8001/health
curl http://localhost:8002/health
```

Each should return `{"status":"ok",...}`. Then open `http://localhost:8000/docs` in your browser for the full interactive API.

## API Walkthrough

### 1. Create some skills

```bash
curl -X POST http://localhost:8000/skills \
  -H "Content-Type: application/json" \
  -d '{"name": "Forklift", "description": "Certified forklift operator"}'

curl -X POST http://localhost:8000/skills \
  -H "Content-Type: application/json" \
  -d '{"name": "First Aid", "description": "Basic first aid certified"}'
```

### 2. Create staff members

```bash
curl -X POST http://localhost:8000/staff \
  -H "Content-Type: application/json" \
  -d '{"name": "Alice Kumar", "email": "alice@example.com", "max_hours_per_week": 40}'

curl -X POST http://localhost:8000/staff \
  -H "Content-Type: application/json" \
  -d '{"name": "Bob Sharma", "email": "bob@example.com", "max_hours_per_week": 40}'
```

### 3. Assign skills to staff

```bash
# Replace the IDs with whatever was returned above
curl -X POST http://localhost:8000/staff/1/skills/1
curl -X POST http://localhost:8000/staff/2/skills/1
curl -X POST http://localhost:8000/staff/2/skills/2
```

### 4. Create a shift

```bash
curl -X POST http://localhost:8000/shifts \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Morning Warehouse",
    "date": "2024-12-15",
    "start_time": "08:00:00",
    "end_time": "16:00:00",
    "required_staff_count": 2,
    "skill_requirements": [{"skill_id": 1, "required_count": 2}]
  }'
```

### 5. Run the scheduler

```bash
curl -X POST http://localhost:8000/shifts/schedule \
  -H "Content-Type: application/json" \
  -d '{"shift_ids": [1]}'
```

You should get back something like:

```json
{
  "success": true,
  "assignments": [
    {"shift_id": 1, "staff_id": 1},
    {"shift_id": 1, "staff_id": 2}
  ],
  "elapsed_ms": 12.4,
  "message": "Scheduled 2 assignments in 12.4 ms"
}
```

At the same time the notification service will log:

```
INFO [app.dispatcher] NOTIFICATION_DISPATCHED {"channel": "log", "event_type": "SHIFT_ASSIGNED", "staff_name": "Alice Kumar", ...}
```

### 6. Trigger the worker manually

```bash
curl -X POST http://localhost:8002/trigger
```

The worker finds every shift currently in `unscheduled` status and passes them all to the solver at once.

## Prometheus Metrics

Open `http://localhost:9090` and try these queries:

| Metric | What it tells you |
|---|---|
| `shiftboard_scheduling_duration_ms` | How long the solver is taking — histogram with buckets from 10 ms to 5 s |
| `shiftboard_scheduling_success_total` | How many scheduling runs ended with a valid assignment set |
| `shiftboard_scheduling_failure_total` | How many runs found no feasible solution |
| `shiftboard_active_staff_total` | Current count of active staff members |
| `notification_events_consumed_total` | Kafka events consumed by the notification service, labelled by event type |
| `notification_notifications_sent_total` | Notifications dispatched, labelled by channel |
| `worker_runs_total` | Worker polling runs labelled by result |
| `worker_unscheduled_shifts_total` | Unscheduled shifts found in the most recent worker run |

## Running Tests Locally

The scheduling-service integration tests need a real PostgreSQL database. The other two services are fully self-contained.

```bash
# Start just the database
docker compose up postgres -d

# Scheduling service — 89% coverage is enforced
cd scheduling-service
pip install -r requirements.txt
TEST_DATABASE_URL=postgresql://shiftboard:shiftboard123@localhost:5432/shiftboard_test \
  alembic upgrade head
pytest tests/ -v --cov=app --cov-report=term-missing --cov-fail-under=89

# Notification service — no external dependencies needed
cd ../notification-service
pip install -r requirements.txt
pytest tests/ -v --cov=app --cov-report=term-missing

# Worker service — all HTTP calls are mocked with respx
cd ../worker-service
pip install -r requirements.txt
pytest tests/ -v --cov=app --cov-report=term-missing
```

## CI/CD Pipeline

Every push to `main` or `develop`, and every pull request targeting `main`, runs three jobs in parallel on GitHub Actions:

**test-scheduling-service** — provisions a real PostgreSQL 15 service container, runs Alembic migrations against it, then runs the full test suite with a hard 89% coverage requirement.

**test-notification-service** — installs dependencies and runs pytest. No external services are needed because all Kafka interaction is tested through mocks.

**test-worker-service** — installs dependencies and runs pytest. Every outbound HTTP call is intercepted by respx so no real scheduling service is required.

## Key Design Decisions

**Backtracking with forward checking rather than a greedy algorithm.** A greedy approach is faster but it can get stuck: once it makes a locally good choice that makes the rest of the problem unsolvable, it has no way to recover. Backtracking guarantees that if a valid assignment exists it will be found, and forward checking keeps it fast enough for realistic inputs.

**Alembic owns schema creation, not the application.** Having `Base.metadata.create_all()` run at application startup is a common FastAPI pattern but it creates a hidden coupling between the app process and the database connection at import time. In production the schema is managed by Alembic migrations run before the server starts. In tests, the test harness creates the schema directly against the test database.

**Manual Kafka offset commits for at-least-once delivery.** The notification consumer commits its offset only after the dispatcher has finished processing the event. If the process crashes mid-dispatch, Kafka will redeliver the message on the next consumer start. The alternative — auto-commit — would silently drop events that were consumed but not yet processed.

**Lazy Kafka producer in the scheduling service.** The producer connection is only established when the first event needs to be sent. This means the scheduling API starts up and serves traffic even when Kafka is not yet available, rather than failing the whole service on startup.

**Three independently deployable services.** If notification volume grows you can scale the notification-service horizontally without touching the scheduling API or the worker. Kafka's consumer group handles the partition distribution automatically.
